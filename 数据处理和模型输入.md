LeRobotR1LiteDataConfig 可以理解成 R1Lite 专用的数据管线模板; 你只要指定 repo_id、root、action_space，它就会自动组装出 OpenPI 训练/推理需要的数据处理流程。


LeRobotR1LiteDataConfig 本身的定义也位于 src/openpi/training/config.py


它主要做三件事：
1）根据 action_space 选择 R1Lite 的输入/输出 transform：
joint_delta -> R1LiteInputs / R1LiteOutputs
abs_eef     -> R1LiteAbsEEFInputs / R1LiteAbsEEFOutputs
delta_eef   -> R1LiteDeltaEEFInputs / R1LiteDeltaEEFOutputs

2）创建数据输入变换 model_transforms：
这里会根据模型类型生成 ResizeImages、TokenizePrompt、PadStatesAndActions 等模型输入处理。

3）调用父类的DataConfigFactory的create_base_config()，补上一些标注的配置参数
repo_id 制作数据集时所设定的repo_id
norm_stats 加载归一化统计数据 默认的会去加载的路径是 ./assets/train_config.name/repo_id/norm_stats.json
root 数据集的位置
prompt_from_task 设置为true
use_quantile_norm 默认为true

#### 2）data_transforms： 作用是把 R1Lite 自己的数据定义，转换成 OpenPI 统一的 observation/action 格式。

R1Lite 里主要有 3 套 Inputs/Outputs，对应 3 种 action space：

joint_delta -> R1LiteInputs / R1LiteOutputs
abs_eef     -> R1LiteAbsEEFInputs / R1LiteAbsEEFOutputs
delta_eef   -> R1LiteDeltaEEFInputs / R1LiteDeltaEEFOutputs

核心区别是：在state的输入提取 和 action的维度确认方便的差别

R1LiteInputs:
  state = 关节状态 14维
  action = 关节增量 14维

R1LiteAbsEEFInputs:
  state = 末端位姿 16维
  action = 绝对末端位姿 16维

R1LiteDeltaEEFInputs:
  state = 末端位姿 16维
  action = 末端位姿增量 14维

以 关节角输入为例：
R1LiteInputs() 主要做：
- 检查相机 key，只允许 head、left_wrist、right_wrist
把图像转成 OpenPI 需要的 image 字段名：
base_0_rgb
left_wrist_0_rgb
right_wrist_0_rgb
- 输入状态
把 R1Lite 原始 53 维 state 压缩成 14 维 joint state：添加到对话
left joint[6] + left gripper[1] + right joint[6] + right gripper[1]
- 标签动作
检查 action 是 14 维 joint_delta

#### 3）Normalize：按 norm stats 归一化 state/actions

作用是把 state 和 actions 缩放到模型更容易学习的数值范围。

它使用 scripts/compute_norm_stats.py 生成的 norm_stats.json。对 pi0.5 默认是 quantile normalize：

(x - q01) / (q99 - q01 + 1e-6) * 2.0 - 1.0
也就是大致把数据映射到 [-1, 1]。

代码在 ./src/openpi/transforms.py line20 class Normalize(DataTransformFn) 类

1）加载norm_stats.json的过程：
- 数据流配置文件继承DataConfigFactory 即LeRobotR1LiteDataConfig(DataConfigFactory)
- DataConfig，会去根据repo_id="r1lite_pack_phone_new_joint_gripper01" 去找：norm_stats.json 并加载，得到
    {
        "state": NormStats(mean=..., std=..., q01=..., q99=...),
        "actions": NormStats(mean=..., std=..., q01=..., q99=...),
    }
加载后的结果存在 data_config.norm_stats



2）它不会处理图片、prompt，只处理有统计量匹配的数值字段。
R1Lite 例子：

归一化前：
    data = {
        "state": np.array([...]),    # 14维
        "actions": np.array([...]),  # 10 x 14
        "image": {...},
        "prompt": "...",
    }
norm_stats 里只有：
    {
        "state": ...,
        "actions": ...,
    }
执行 Normalize 后：

    data = {
        "state": normalized_state,
        "actions": normalized_actions,
        "image": {...},      # 不动
        "prompt": "...",     # 不动
    }
所以它不会处理图片、prompt，只处理有统计量匹配的数值字段。

#### 4）model_transforms：模型专用加工；作用是把已经归一化的数据，变成具体模型需要的输入形式。

对你的 pi05 配置，model_transforms 包含：

- InjectDefaultPrompt # 如果数据没有 prompt，就注入默认任务 prompt
- ResizeImages(224, 224) # 图像 resize 到 224x224
- TokenizePrompt(..., discrete_state_input=True) # 把 prompt tokenize  如果 discrete_state_input=True，把归一化后的 state 离散成数字并拼进 prompt
- PadStatesAndActions(model_config.action_dim) # 把 state/actions  padding 到模型的 action_dim openpi默认是32


离散化的state的意义是：

1 pi0.5与pi0明确说明了两点不同：Pi05 has two differences from Pi0:
1）明确使用离散的state输入 - the state input is part of the discrete language tokens rather than a continuous input that is part of the suffix
2）动作expert有个不一样 - the action expert uses adaRMSNorm to inject the flow matching timestep

R1LiteInputs 会把原始 53 维机器人状态压缩成 14 维
然后 Normalize 把这个 14 维 state 归一化。假设归一化后是：
state_14 = [0.1, -0.3, 0.7, ..., 0.2]   # 共 14 个数

当 discrete_state_input=True 时，TokenizePrompt 会先读取当前的 data["state"]。这时候 state 还没有 padding，所以还是 14 维：
然后 tokenizer 会把这 14 个数离散化成整数 token，比如：[140, 89, 216, ..., 153]
再送入到prompt：
Task: pack phone..., State: 140 89 216 ... 153;
Action:

然后才执行 PadStatesAndActions(32)，把 data["state"] 从 14 维补零到 32 维：[0.1, -0.3, 0.7, ..., 0.2, 0, 0, ..., 0] 这个 32 维的 state_32 会继续保存在 observation.state 里，作为模型输入结构的一部分。

所以最终模型看到的是两份 state 信息：

1. tokenized_prompt 里：
   14 维 state 被离散成数字字符串，拼进语言 token

2. observation.state 里：
   14 维 state 被补零成 32 维 float 向量


但对于 pi05=True 的模型来说，核心使用的是第 1 份：prompt 里的离散 state token。因为在 Pi0.embed_suffix() 里，pi0.5 不会再把 observation.state 通过 state_proj 加成 continuous state token。

    if not self.pi05:
        state_token = self.state_proj(obs.state)[:, None, :]

pi0:
  使用 observation.state -> state_proj -> continuous state token

pi0.5:
  不走 state_proj
  state 已经通过 TokenizePrompt 进入 tokenized_prompt
